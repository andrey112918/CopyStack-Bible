## Motion Plugin Registry

Implemented a centralized registry responsible for:

- plugin discovery
- plugin lookup
- motion type registration
- renderer integration

The renderer no longer needs hardcoded motion logic.

---

## Motion Profiles

Created configurable motion profiles.

Each plugin now supports:

- low
- medium
- high

instead of hardcoded values.

This allows AI Directors to select cinematic intensity dynamically.

---

## Motion Renderer

Implemented MotionRenderer.

Responsibilities:

- receive MotionInstructions
- resolve plugins
- build FilterGraph
- generate FFmpeg filter chain

## Filter Graph

Implemented an abstraction layer over FFmpeg.

New components:

- FilterGraph
- FilterOperation
- FilterGraphBuilder

Instead of concatenating strings everywhere,
filters are now represented as objects.

This is the foundation for future rendering backends.

# Architectural Discoveries

During real FFmpeg rendering several important limitations were discovered.

## Discovery 1

scale expressions require:

eval=frame

when using timeline variables.

---

## Discovery 2

crop filter does not support:

enable=

This makes timeline-controlled crop animations unsuitable for several motion effects.

---

## Discovery 3

Direct generation of FFmpeg expressions inside plugins does not scale.

Future architecture should separate:

Motion Decision

↓

Transform Model

↓

Renderer Backend

↓

FFmpeg

instead of:

Plugin

↓

FFmpeg String

---

# Architectural Decision

Motion Plugin System V1 is considered complete.

Current backend remains FFmpeg.

Future work will introduce a Transform Engine capable of supporting:

- FFmpeg
- GPU rendering
- OpenGL
- Vulkan
- future render backends

without changing AI logic.
